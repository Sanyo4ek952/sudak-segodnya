# AGENTS.md

# Проект: «Судак Сегодня»

## Назначение

Этот файл содержит обязательные правила для Codex и других AI-агентов, работающих с проектом.

Перед любой задачей агент обязан:

1. Прочитать `AGENTS.md`.
2. Изучить относящиеся к задаче документы: `Product.md`, `Screen.md`, `DesignSystem.md`.
3. Осмотреть существующую архитектуру, похожие экраны и компоненты.
4. Определить минимальную область изменений.
5. Не добавлять функции, которых нет в документации или текущем задании.

При конфликте между предположением агента и документацией приоритет имеют документация и явное задание пользователя.

---

## Контекст продукта

«Судак Сегодня» — mobile-first веб-приложение с единой городской лентой актуальной информации для жителей и туристов Судака.

Главная ценность:

- быстро показать, что происходит сейчас и в ближайшее время;
- дать организациям возможность самостоятельно публиковать актуальную информацию;
- не превращать приложение в перегруженный каталог, социальную сеть или маркетплейс.

Основной интерфейс — публичная городская лента.

## Архитектура

Использовать Feature-Sliced Design без избыточного дробления.

Уровни:

- `app` — маршруты, layouts, providers, конфигурация;
- `widgets` — крупные составные блоки страниц;
- `features` — законченные пользовательские действия;
- `entities` — бизнес-сущности;
- `shared` — UI, утилиты, API-клиенты и инфраструктура.

Правила:

- Server Components использовать по умолчанию.
- Client Components добавлять только при необходимости интерактивности или браузерного API.
- Не размещать бизнес-логику в `shared/ui`.
- Защищённые операции выполнять на сервере.
- Права доступа проверять на сервере и через Supabase RLS.
- Не передавать service role key и другие секреты в браузер.
- Входные данные валидировать через Zod.
- Не создавать абстракции «на будущее».
- Не проводить массовый рефакторинг вне границ задачи.
- Не менять архитектуру без явного обоснования.

## Правила продукта

- Главная страница должна быть лёгкой и быстро показывать ленту.
- Погода отображается компактно.
- Подробный прогноз открывается отдельно или по нажатию.
- Закреплённое важное объявление показывается только при наличии.
- Основную часть главной занимает городская лента.
- Логотип организации необязателен.
- При отсутствии логотипа использовать основную фотографию или нейтральную заглушку.
- Завершённые события автоматически скрываются из основной ленты.
- Отменённые события имеют понятный статус.
- Кнопка «Написать» на странице публикации не используется.
- «Сообщить о неточности» размещается в дополнительном меню.
- Одна жалоба не скрывает публикацию автоматически.
- Заказы, оплата, комментарии, отзывы, чат и сложные рекомендации не входят в MVP.
- Не добавлять функции вне документации без отдельного задания.

## Единый визуальный стиль

Любой новый экран или компонент должен соответствовать существующему стилю приложения.

Перед реализацией UI агент обязан:

1. Найти похожие экраны.
2. Найти существующие компоненты.
3. Проверить правила и токены в `DesignSystem.md`.
4. Перечислить, что будет переиспользовано.
5. Только после этого создавать новые компоненты.

Запрещено:

- придумывать отдельный визуальный стиль внутри одной задачи;
- использовать случайные цвета, радиусы, тени и отступы;
- дублировать существующие UI-компоненты;
- копировать JSX существующего компонента ради небольшого отличия;
- менять глобальные стили ради одной страницы;
- добавлять новую UI-библиотеку без явного решения.

## Design tokens

Базовые значения должны храниться централизованно:

- цвета;
- поверхности;
- цвета текста;
- границы;
- радиусы;
- тени;
- типографика;
- отступы;
- размеры контейнеров;
- состояния `success`, `warning`, `error`, `info`.

Не использовать произвольные Tailwind-значения, если нужное значение уже определено токеном или существующим паттерном.

Новый токен добавляется только тогда, когда он системный и будет использоваться повторно.

## Переиспользование компонентов

Перед созданием нового компонента обязательно:

1. Проверить `shared/ui`.
2. Проверить компоненты текущей сущности.
3. Проверить компоненты текущей фичи и виджета.
4. Проверить, можно ли расширить существующий компонент через `props`, `variants`, композицию, слоты или `children`.

Не дублировать:

- кнопки;
- поля ввода;
- карточки;
- модальные окна;
- бейджи;
- пустые состояния;
- загрузчики;
- элементы навигации;
- информационные блоки;
- одинаковые формы.

Новый общий компонент создаётся, когда:

- он нужен минимум в двух местах;
- у него понятный публичный API;
- он не зависит от одной страницы;
- переиспользование реально уменьшает дублирование.

Компонент, используемый только внутри одного экрана, должен оставаться рядом с соответствующим виджетом или фичей.

## Изменение общего компонента

Перед изменением переиспользуемого компонента:

1. Найти все места использования.
2. Оценить возможные регрессии.
3. Сохранить обратную совместимость, если её изменение не является частью задачи.
4. Не добавлять большое количество несвязанных boolean-props.
5. Предпочитать variants и композицию.
6. Проверить mobile и desktop.

---

## Mobile-first и доступность

Обязательно:

- проверять узкий мобильный экран;
- не допускать горизонтального переполнения;
- использовать удобные области нажатия;
- учитывать bottom navigation и safe area;
- использовать семантический HTML;
- добавлять подписи кнопкам только с иконкой;
- не передавать смысл только цветом;
- поддерживать loading, empty и error states.

## Проверка изменений

Проверки должны соответствовать масштабу изменения.

Не запускать автоматически для каждой задачи:

- все тесты проекта;
- полный `build`;
- полный `lint`;
- все end-to-end тесты.

Сначала определить:

- какие файлы изменены;
- какие сценарии затронуты;
- какие минимальные проверки подтверждают корректность;
- требуется ли расширенная проверка.

### Небольшая локальная правка

Для текста, отступа, локального стиля, простого условия или одного локального компонента достаточно:

- проверить затронутые файлы;
- запустить связанный тест, если он существует;
- выполнить локальный lint или type-check, если возможно;
- проверить соответствующий экран.

Полный `build` обычно не нужен.

### Изменение переиспользуемого компонента

Нужно:

- найти все места использования;
- запустить тесты компонента и связанных сценариев;
- проверить основные variants и состояния;
- проверить мобильное представление.

### Когда нужны широкие проверки

Расширенные или полные проверки нужны при изменении:

- `shared/ui`;
- авторизации;
- ролей и прав доступа;
- RLS;
- схемы базы данных;
- API-контрактов;
- глобальной конфигурации;
- зависимостей;
- маршрутизации;
- критической бизнес-логики;
- окружения и сборки;
- большого числа связанных модулей;
- кода перед релизом.

### Новые тесты

Не писать тесты только ради покрытия.

Добавлять тесты, если задача затрагивает:

- важное бизнес-правило;
- права доступа;
- преобразование данных;
- пограничные случаи;
- поведение, которое легко сломать повторно.

Для простой визуальной правки новый тест обычно не требуется.

## Экономия токенов и времени

Агент должен работать экономно, но не в ущерб результату.

Нужно:

- читать только относящиеся к задаче файлы;
- не исследовать весь репозиторий без причины;
- не повторять известный контекст;
- не генерировать длинные отчёты;
- не запускать ненужные команды;
- не переписывать рабочий код без необходимости;
- не создавать лишние тесты и документацию;
- не трогать файлы вне задачи.

Экономия не является основанием пропускать проверки безопасности, прав доступа или критической бизнес-логики.

---

## Порядок выполнения задачи

Перед изменением кода кратко указать:

- как понята задача;
- какие файлы и области будут затронуты;
- какие компоненты будут переиспользованы;
- требуется ли новый компонент;
- какие проверки будут выполнены.

После задачи кратко указать:

- что изменено;
- какие компоненты переиспользованы;
- какие компоненты созданы и почему;
- какие проверки запущены;
- что не запускалось и почему;
- известные ограничения или риски.

Для маленькой задачи отчёт должен быть коротким.

## PWA и offline-режим

- Проект разрабатывается как mobile-first PWA.
- Все новые экраны должны использовать единый `AppShell`.
- В standalone-режиме обязательно учитывать `viewport-fit=cover`, верхний и нижний safe-area.
- Защищённые данные не кешируются в Cache API.
- Offline-режим не должен показывать устаревшие административные данные или данные кабинета организации.
- Не кешировать `/admin/**`, `/business/**`, auth routes, API, Supabase API/Auth/Storage, Server Actions, RPC, формы и мутации.
- Новые функции проверять в обычной вкладке и standalone-режиме.
- Подробные правила PWA описаны в `docs/PWA.md`.
