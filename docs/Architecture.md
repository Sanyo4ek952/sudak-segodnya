# Architecture.md

# «Судак Сегодня» — backend-архитектура MVP

## 1. Цель backend

Backend MVP должен заменить моковые данные публичного прототипа на Supabase, не меняя продуктовую суть приложения:

- публичная городская лента читается без регистрации;
- организации сами ведут профиль, публикации, меню или услуги;
- администратор проверяет заявки, модерирует спорные данные и управляет важными объявлениями;
- завершённые материалы не попадают в основную ленту;
- заказы, оплата, комментарии, отзывы, чат и сложные рекомендации не добавляются.

Основной стек:

- Next.js App Router;
- Server Components для публичного чтения по умолчанию;
- Server Actions и Route Handlers для защищённых операций;
- Supabase PostgreSQL;
- Supabase Auth;
- Supabase Storage;
- Supabase RLS;
- Zod для входной валидации.

## 2. Текущая база приложения

Текущий проект уже разделён по Feature-Sliced Design:

- `app` — маршруты публичной части;
- `widgets` — лента, навигация, погода, избранное;
- `features` — локальное избранное;
- `entities` — организации и публикации;
- `shared` — UI, утилиты, конфигурация.

Сейчас публичные экраны используют моковые данные из `entities/*/model/mock.ts`. Backend должен сначала повторить текущую публичную модель данных, затем открыть кабинет организации и админ-панель.

## 3. Роли

### Публичный пользователь

Не авторизуется и может:

- читать публичную ленту;
- открывать опубликованные и актуальные публикации;
- открывать активные организации;
- читать доступные меню или услуги;
- отправить сообщение о неточности с ограничением частоты;
- сохранять избранное локально в браузере.

Публичный пользователь не получает аккаунт в MVP и не имеет серверного избранного.

### Представитель организации

Авторизуется через Supabase Auth и может:

- подать заявку на организацию;
- видеть свои заявки;
- управлять только организациями, где есть активная запись в `organization_members`;
- редактировать профиль своей организации;
- создавать, редактировать, отменять и завершать свои публикации;
- управлять меню или услугами своей организации;
- смотреть простую статистику своей организации.

Представитель не может менять чужие организации, подменять `organization_id`, подменять `author_id`, одобрять заявки, скрывать публикации административным статусом или блокировать контент.

### Администратор

Авторизуется через Supabase Auth и имеет `profiles.role = admin`.

Администратор может:

- просматривать пользователей и профили;
- проверять заявки организаций;
- одобрять, отклонять и запрашивать уточнения;
- активировать, блокировать и восстанавливать организации;
- скрывать и восстанавливать публикации;
- обрабатывать сообщения о неточности;
- создавать и закреплять важные объявления;
- смотреть базовую аналитику.

## 4. Публичное чтение

Публичное чтение выполняется anon-клиентом Supabase и ограничивается RLS.

Публично доступны только:

- организации со статусом `active`;
- опубликованные или отменённые публикации активных организаций, если они ещё актуальны;
- завершённые публикации только на странице организации или в истории, если такой экран будет явно реализован;
- активные важные объявления в пределах периода показа;
- активные типы организаций и активные категории публикаций;
- доступные категории и позиции меню активных организаций.

Основная лента должна исключать:

- `draft`;
- `scheduled`, если время публикации ещё не наступило;
- `moderation`;
- `completed`;
- `hidden`;
- `blocked`;
- публикации заблокированных организаций;
- публикации с истёкшим `ends_at` или `valid_until`, кроме отменённых с ещё полезным сроком показа.

## 5. Авторизация

Supabase Auth используется только для представителей организаций и администраторов.

Рекомендуемый поток:

1. Пользователь регистрируется или входит.
2. После регистрации создаётся строка `profiles` с ролью `user`.
3. Пользователь подаёт заявку на организацию.
4. После одобрения создаётся или активируется организация и запись `organization_members`.
5. Администратор назначается вручную через защищённую операцию или SQL на этапе запуска проекта.

Обычный публичный просмотр не требует регистрации.

## 6. Кабинет организации

Кабинет строится поверх membership-проверки:

- пользователь считается представителем организации, если есть запись `organization_members.user_id = auth.uid()`, `organization_id = ...`, `is_active = true`;
- все запросы кабинета фильтруются по организациям текущего пользователя;
- формы дополнительно валидируются Zod-схемами на сервере.

Разделы MVP:

- обзор: статус организации, активные публикации, ближайшие события, базовые показатели;
- профиль: название, основной тип организации, описание, адрес, контакты, изображения, график;
- публикации: создание, черновик, планирование, публикация, отмена, завершение, удаление черновика;
- меню или услуги: категории, позиции, доступность, порядок;
- статистика: агрегированные события по публикациям и организации;
- настройки: представители организации.

## 7. Админ-панель

Админ-панель использует отдельные защищённые серверные операции. Доступ проверяется на сервере и RLS.

Разделы MVP:

- заявки организаций;
- организации;
- публикации;
- сообщения о неточностях;
- важные объявления;
- пользователи.

Административные действия выполняются через Server Actions или RPC, чтобы не открывать прямое изменение служебных статусов из браузера.

## 8. Серверные операции

Protected mutations должны выполняться на сервере:

- создание и обновление заявки организации;
- одобрение или отклонение заявки;
- создание и обновление организации;
- создание и обновление публикации;
- смена статуса публикации;
- загрузка и привязка изображений;
- создание и обработка сообщений о неточности;
- создание и снятие важных объявлений;
- запись аналитических событий, если событие нельзя безопасно писать напрямую anon-клиентом.

Правила:

- входные данные валидируются через Zod;
- `user_id`, `author_id`, `reviewed_by`, `created_by` берутся из серверного контекста, а не из тела запроса;
- `organization_id` проверяется по `organization_members`;
- административные статусы меняются только администратором;
- service role key используется только в серверном окружении для точечных операций, где RLS недостаточно, и не заменяет обычные RLS-политики.

Для публичной отправки сообщений о неточности используется RPC `create_inaccuracy_report`. Он проверяет, что публикация публична, нормализует причину и комментарий, фиксирует `auth.uid()` для авторизованного пользователя или серверный cookie-fingerprint для гостя, ограничивает повторные сообщения по одному объекту и причине, а также ограничивает частоту отправки.

## 9. Изображения

Изображения хранятся в Supabase Storage, а прикладная привязка хранится в `media_assets`.

Минимальные buckets:

- `organization-images` — логотипы и обложки организаций;
- `publication-images` — изображения публикаций;
- `menu-images` — изображения позиций меню или услуг;
- `application-confirmation-images` — приватные подтверждающие фото заявок.

Рекомендуемые пути:

- `organizations/{organization_id}/logo/{file_id}`;
- `organizations/{organization_id}/cover/{file_id}`;
- `publications/{publication_id}/{file_id}`;
- `menu-items/{menu_item_id}/{file_id}`;
- `applications/{application_id}/confirmation/{file_id}`.

Назначение фото фиксируется через `media_assets.purpose`:

- `organization_logo`;
- `organization_cover`;
- `application_confirmation`;
- `publication_photo`;
- `menu_item_photo`.

Чтение:

- публичное чтение разрешено только для `media_assets.visibility = public`, связанных с публично доступными активными организациями, опубликованными публикациями или доступными позициями меню;
- подтверждающие фото заявок приватны и доступны только заявителю и администратору;
- buckets остаются private, а доступ к файлам выдаётся через серверную операцию или Storage policy на основе `media_assets`.

Запись:

- представитель может загружать файлы только для своей организации;
- загрузка выполняется через серверную операцию, которая проверяет membership;
- администратор может загружать изображения важных объявлений и править изображения организаций при модерации.

Удаление:

- физическое удаление файлов выполняется серверной операцией после отвязки записи;
- при удалении черновика можно удалять неиспользуемые файлы;
- опубликованные материалы лучше переводить в скрытое состояние, чтобы не ломать публичные ссылки.

## 10. Автоматические процессы

Для MVP нужны только простые фоновые процессы:

- автоматическое завершение публикаций с истёкшим `ends_at` или `valid_until`;
- перевод важных объявлений в `expired` после `active_until`;
- базовая агрегация аналитики может выполняться запросами по `analytics_events`, без отдельного хранилища.

Автоматическое завершение можно реализовать как Supabase Scheduled Function, cron-triggered RPC или серверную задачу на Vercel. Даже до внедрения job публичные выборки должны фильтровать истёкшие публикации по времени.

## 11. Границы MVP

Не включать в backend MVP:

- заказы;
- оплату;
- билеты;
- бронирование;
- публичные комментарии;
- отзывы и рейтинги;
- чат;
- программу лояльности;
- сложные рекомендации;
- серверную синхронизацию избранного.
## PWA и offline-безопасность

Проект развивается как mobile-first PWA без изменения backend-контрактов MVP.

Правила:

- все новые экраны используют единый `AppShell`;
- protected routes, auth routes, Server Actions, RPC, Supabase API/Auth/Storage и ответы с приватными признаками не попадают в Cache API;
- offline-режим не показывает устаревшие административные данные или данные кабинета организации;
- публичная навигация может иметь только безопасный network-first fallback;
- новые функции проверяются в standalone-режиме и на мобильных ширинах.

Погодные данные для публичного MVP запрашиваются сервером из Open-Meteo с короткой revalidate-стратегией Next.js. Это не заменяет offline-кеш: `/weather`, `/api/**` и актуальные погодные ответы не должны сохраняться Service Worker.

Подробные правила описаны в `docs/PWA.md`.
