# «Судак Сегодня» — правила дизайн-системы

## 1. Цель

Дизайн-система нужна, чтобы приложение сохраняло единый стиль, а новые экраны и функции не превращались в набор несвязанных решений.

Она должна обеспечивать:

- единый визуальный язык;
- повторное использование компонентов;
- понятную структуру интерфейса;
- стабильное развитие проекта;
- удобство на мобильных устройствах;
- минимальное количество локальных и случайных стилей.

Первое визуальное направление публичного прототипа утверждено как спокойный городской стиль: светлый фон, белые карточки, морской основной акцент и сдержанные семантические статусы.

---

## 2. Основные принципы

### Mobile-first

Основной сценарий использования — телефон. Сначала проектируется мобильная версия, затем планшет и desktop.

### Контент важнее оформления

Интерфейс должен помогать быстро увидеть:

- что происходит;
- когда;
- где;
- сколько стоит;
- кто организатор;
- как связаться или построить маршрут.

### Спокойный городской стиль

Интерфейс должен выглядеть современно, понятно и аккуратно, без визуального шума.

Не использовать без необходимости:

- много акцентных цветов;
- тяжёлые градиенты;
- сильные тени;
- декоративные анимации;
- разные стили одинаковых карточек.

### Единообразие

Одинаковые действия и данные должны выглядеть одинаково на всех страницах.

---

## 3. Design tokens

Базовые значения задаются централизованно через CSS variables и Tailwind.

### Цвета

Использовать токены:

- `background`: `#f6f9fa`;
- `surface`: `#ffffff`;
- `surface-muted`: `#e5eef1`;
- `foreground`: `#1c2b36`;
- `foreground-muted`: `#606f7b`;
- `border`: `#ced9de`;
- `primary`: `#166d88`;
- `primary-foreground`: `#ffffff`;
- `secondary`: `#f7eacb`;
- `accent`: `#42947e`;
- `success`: `#31865c`;
- `warning`: `#d7840d`;
- `error`: `#cc3329`;
- `info`: `#2c77b5`.

Правила:

- один основной акцентный цвет;
- семантические цвета только для статусов;
- никаких случайных hex-значений внутри компонентов;
- не передавать смысл только цветом.

### Радиусы

Ограниченный набор:

- `sm`: `6px`;
- `md`: `8px`;
- `lg`: `12px`;
- `xl`: `16px`;
- `full`: `9999px`.

### Тени

Допустимый набор:

- без тени;
- лёгкая тень карточки: `0 8px 28px hsl(205 32% 16% / 0.08)`;
- тень всплывающего слоя: `0 18px 50px hsl(205 32% 16% / 0.16)`.

### Отступы

Использовать системную шкалу Tailwind. Произвольные значения — только при обоснованной необходимости.

Основные значения публичной части:

- горизонтальный отступ мобильного контейнера: `16px`;
- горизонтальный отступ `sm`: `24px`;
- горизонтальный отступ `lg`: `32px`;
- вертикальный отступ между основными секциями: `24px`;
- сеточный `gap` карточек: `16px`;
- внутренний отступ карточки: `16px` на мобильном и `20px` с `sm`;
- нижний отступ `main` на мобильном: `112px`, чтобы контент не перекрывался нижней навигацией.

### Типографика

Уровни:

- заголовок страницы: `30px / 37.5px` на мобильном, `36px / 40px` с `sm`, `font-weight: 600`;
- заголовок секции: `20px / 28px`, `font-weight: 600`;
- заголовок карточки: `18px / 28px`, `font-weight: 600`;
- основной текст: `16px / 28px`;
- вторичный текст: `14px / 24px`;
- подпись и бейдж: `12px / 16px`;
- крупное значение: `36px / 40px`, `font-weight: 600`.

### Контейнеры

Централизованно определить:

- максимальную ширину публичного контента: `72rem`;
- ширину форм и компактных состояний: `42rem`;
- ширину страницы публикации: `48rem`;
- ширину страницы организации: `56rem`;
- мобильные горизонтальные отступы: `16px`;
- safe-area для нижней навигации: `env(safe-area-inset-bottom) + 8px`.

---

## 4. Типографика

Каждая страница должна иметь понятную иерархию:

1. Заголовок страницы.
2. Ключевая информация.
3. Заголовки секций.
4. Основной текст.
5. Вторичные данные.
6. Подписи.

Правила:

- не использовать слишком много размеров;
- не делать весь текст полужирным;
- не уменьшать текст ради размещения большего количества информации;
- дата, время, цена и статус должны быстро находиться взглядом;
- вторичная информация не конкурирует с названием.

---

## 5. Сетка и отступы

### Мобильная версия

- единые горизонтальные отступы;
- комфортные расстояния между секциями;
- карточки не слипаются;
- интерактивные элементы имеют удобную область нажатия;
- нижняя навигация не перекрывает контент.

### Desktop

- ограниченный контейнер;
- дополнительные колонки только при реальной пользе;
- сохраняется мобильная визуальная иерархия.

---

## 6. Базовые компоненты

Общие компоненты размещаются в `shared/ui`.

Базовый набор:

- `Button`;
- `IconButton`;
- `Input`;
- `Textarea`;
- `Select`;
- `Checkbox`;
- `RadioGroup`;
- `Switch`;
- `Badge`;
- `Card`;
- `Dialog`;
- `Drawer`;
- `DropdownMenu`;
- `Tabs`;
- `Skeleton`;
- `Spinner`;
- `EmptyState`;
- `ErrorState`;
- `Pagination` или `LoadMore`;
- `OrganizationImage`;
- `FormField`;
- `SectionHeader`.

Перед созданием нового компонента обязательно проверить существующие.

---

## 7. Кнопки

Variants:

- `primary`;
- `secondary`;
- `outline`;
- `ghost`;
- `destructive`;
- `link`.

Размеры:

- `sm`;
- `md`;
- `lg`;
- `icon`.

Правила:

- одна главная primary-кнопка на локальный блок;
- вторичные действия не конкурируют с главным;
- destructive-действия визуально отделяются;
- кнопки только с иконкой имеют доступную подпись;
- не создавать новый цвет кнопки для каждой фичи.

---

## 8. Карточки

### Общая карточка

Карточки используют единые фон, границу, радиус, внутренние отступы и состояния.

### Карточка публикации

Приоритет:

1. Статус.
2. Название.
3. Дата и время.
4. Место.
5. Цена.
6. Организация.
7. Изображение и вторичное описание.

Не добавлять много кнопок внутрь карточки.

### Карточка организации

Показывает:

- изображение;
- название;
- категорию;
- адрес;
- статус работы при достоверных данных;
- активные публикации.

---

## 9. Изображения организаций

Логотип необязателен.

Порядок отображения:

1. Логотип.
2. Основная фотография.
3. Нейтральная заглушка.

Заглушка может содержать первую букву названия и не должна выглядеть как ошибка.

---

## 10. Статусы и бейджи

Использовать единый `Badge`.

Примеры:

- Сейчас;
- Сегодня;
- Завтра;
- Бесплатно;
- Регулярно;
- Отменено;
- Завершено;
- Черновик;
- На модерации.

Одинаковые статусы всегда выглядят одинаково.

---

## 11. Формы

- логически группировать поля;
- не показывать длинную форму одним сплошным блоком;
- показывать ошибки рядом с полем;
- сохранять введённые данные при ошибке;
- предупреждать о несохранённых изменениях;
- использовать видимые label;
- проверять формы на мобильном экране.

Создание публикации делится на этапы:

1. Тип.
2. Содержание.
3. Дата и актуальность.
4. Место и цена.
5. Предпросмотр.
6. Публикация.

---

## 12. Навигация

### Публичная часть

На мобильном — нижняя навигация не более чем из пяти пунктов.

### Кабинет организации

На мобильном не использовать постоянно раскрытое широкое боковое меню.

Подходящие варианты:

- вкладки;
- верхнее меню;
- drawer;
- адаптивная боковая навигация на desktop.

---

## 13. Модальные окна и drawers

На мобильном для большого содержимого предпочтителен `Drawer`.

`Dialog` использовать для:

- подтверждения;
- небольшой формы;
- предупреждения;
- короткого действия.

---

## 14. Состояния интерфейса

Каждый экран и компонент данных должен поддерживать:

- loading;
- empty;
- error;
- success;
- disabled;
- unavailable.

### Loading

- skeleton для контента;
- spinner для короткого локального действия;
- не блокировать весь экран без необходимости.

### Empty

Показать, почему список пуст и что пользователь может сделать дальше.

### Error

Понятный текст и действие «Повторить», без технических деталей.

---

## 15. Адаптивность

Проверять:

- узкий телефон;
- обычный телефон;
- планшет;
- desktop.

Не допускать:

- горизонтальный скролл;
- кнопки за пределами экрана;
- перекрытие нижней навигацией;
- слишком плотные таблицы;
- мелкие области нажатия.

---

## 16. Доступность

- семантический HTML;
- видимый focus;
- управление с клавиатуры;
- `aria-label` для кнопок только с иконкой;
- связь label с полями;
- достаточный контраст;
- не использовать цвет как единственный сигнал;
- учитывать `prefers-reduced-motion`.

---

## 17. Анимации

Разрешены только функциональные анимации:

- открытие drawer;
- появление меню;
- смена состояния;
- skeleton;
- мягкая обратная связь.

---

## 18. Правила создания нового компонента

Перед созданием:

1. Проверить `shared/ui`.
2. Проверить компоненты сущности.
3. Проверить `features` и `widgets`.
4. Проверить расширение через props, variants и композицию.
5. Определить, общий компонент или локальный.

Новый общий компонент создаётся, если:

- нужен минимум в двух местах;
- имеет понятный API;
- не зависит от одной страницы;
- реально уменьшает дублирование.

---

## 19. Первое визуальное направление

Перед окончательной фиксацией палитры и токенов нужно утвердить:

1. Мобильную главную ленту.
2. Карточку публикации.
3. Страницу публикации.
4. Карточку организации.
5. Базовую навигацию.

Текущий публичный прототип фиксирует это направление. Новые публичные экраны должны использовать токены из раздела `Design tokens`, а изменения палитры, радиусов, типографики и контейнеров выполняются только как системное решение.
## PWA и safe-area

Приложение проектируется как mobile-first PWA и должно корректно работать в standalone-режиме.

Правила:

- верхняя панель учитывает `env(safe-area-inset-top)`;
- нижняя навигация и всплывающие панели учитывают `env(safe-area-inset-bottom)`;
- контент не перекрывается системными областями и bottom navigation;
- offline-страница использует общий `AppShell`, `SectionHeader`, `Card` и `Button`;
- install/update UI использует существующие компоненты и design tokens;
- новые экраны проверяются в обычной вкладке, standalone-режиме, на 320, 375, 430 px и desktop.
