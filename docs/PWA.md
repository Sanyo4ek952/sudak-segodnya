# «Судак Сегодня» — правила PWA и offline-режима

## 1. Цель

Приложение развивается как mobile-first PWA: его можно установить на устройство, открыть в standalone-режиме и получить безопасную минимальную offline-страницу при отсутствии сети.

PWA-режим не меняет продуктовую модель MVP:

- не добавляет push-уведомления;
- не добавляет background sync;
- не добавляет offline-создание или редактирование данных;
- не кеширует Supabase-данные, погоду, административные данные и данные кабинета организации.

## 2. Manifest и иконки

Manifest реализуется через App Router в `src/app/manifest.ts`.

Обязательные поля:

- `id`;
- `name`;
- `short_name`;
- `description`;
- `lang: "ru"`;
- `start_url: "/"`;
- `scope: "/"`;
- `display: "standalone"`;
- `background_color`;
- `theme_color`;
- категории;
- обычные и maskable-иконки.

Текущие иконки в `public/` являются временными нейтральными иконками на основе дизайн-токенов проекта. После утверждения бренда их нужно заменить без изменения PWA-контракта.

## 3. AppShell и standalone

Все новые экраны используют единый `AppShell`.

Обязательные правила:

- учитывать `viewport-fit=cover`;
- учитывать `env(safe-area-inset-top)` для верхней панели;
- учитывать `env(safe-area-inset-bottom)` для нижней навигации и всплывающих PWA-панелей;
- не создавать вторую навигацию на offline-странице;
- проверять новые экраны в обычной вкладке и standalone-режиме.

## 4. Service worker

Service worker находится в `public/sw.js` и регистрируется только:

- в production;
- или при явном `NEXT_PUBLIC_ENABLE_PWA_SW=true` для локальной проверки.

Регистрация не должна ломать приложение при ошибке. Обновление service worker показывается как компактное сообщение и применяется только после действия пользователя, чтобы не уничтожать введённые в форму данные.

Имена кешей должны быть версионированными. Старые версии кешей удаляются при `activate`.

## 5. Стратегия кеширования

Кешировать можно только безопасные ресурсы:

- `/_next/static/**`;
- favicon и PWA-иконки;
- `/offline`;
- ранее успешно загруженные публичные HTML-навигации, если ответ не содержит приватных признаков.

Для публичной навигации используется network-first:

1. сначала сеть;
2. при offline — ранее сохранённая публичная страница;
3. если её нет — `/offline`.

Offline-режим не гарантирует актуальность публикаций, погоды, статусов мероприятий или организаций.

## 6. Никогда не кешировать

Полностью исключены из Cache API:

- `/admin/**`;
- `/business/**`;
- `/login`;
- `/register`;
- `/forgot-password`;
- `/reset-password`;
- `/weather`;
- `/auth/**`;
- `/api/**`;
- Supabase API, Auth и Storage;
- запросы с `Authorization`;
- ответы с `Set-Cookie`;
- ответы с `Cache-Control: no-store` или `private`;
- все методы, кроме `GET`;
- Server Actions, RPC, формы и мутации.

Если сеть недоступна для защищённого маршрута, пользователь должен увидеть сетевую ошибку или безопасный публичный fallback, но не устаревшие приватные данные.

## 7. Install experience

Chromium:

- использовать `beforeinstallprompt`;
- не показывать кнопку, если приложение установлено или установка не поддерживается;
- позволять скрыть предложение;
- не показывать повторно при каждом переходе.

iOS:

- показывать короткую инструкцию через меню «Поделиться» → «На экран Домой»;
- показывать только вне standalone-режима.

Install UI не показывается на защищённых, auth и формовых маршрутах.

## 8. Проверки

Перед релизом PWA-изменений выполнить:

- `npx tsc --noEmit`;
- `npm run lint`;
- `npm run build`;
- проверку manifest и icon paths;
- проверку регистрации service worker на production build или при `NEXT_PUBLIC_ENABLE_PWA_SW=true`;
- проверку offline fallback;
- проверку установки в Chromium и standalone display;
- проверку ширин 320, 375, 430 px и desktop;
- проверку Cache Storage после посещения `/`, `/organizations`, `/business`, `/admin`.

Запрещено отправлять изменения, если в Cache Storage обнаружены protected routes, Supabase-ответы, auth-страницы или приватные HTML-ответы.
