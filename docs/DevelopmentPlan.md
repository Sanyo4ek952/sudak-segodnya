# DevelopmentPlan.md

# «Судак Сегодня» — план backend-разработки MVP

## 1. Принципы реализации

- Идти маленькими этапами.
- Не добавлять функции вне MVP.
- Сохранять текущую FSD-структуру проекта.
- Публичные экраны сначала должны выглядеть так же, как текущий прототип на моковых данных.
- Server Components использовать для чтения по умолчанию.
- Защищённые операции выполнять через Server Actions или Route Handlers.
- Валидацию форм делать через Zod.
- Проверки подбирать по масштабу этапа, не запускать полный build/lint/test без необходимости.

## 2. Этап 1. Подготовка Supabase-контрактов

Цель: подготовить приложение к Supabase без бизнес-логики.

Статус: выполнено 2026-07-21. Supabase-зависимости добавлены, `.env`-контракт описан через публичные переменные, созданы browser/server clients и базовые типы данных. Публичный UI на момент этапа оставался на моковых данных, а service role key в клиентском коде не используется.

Работы:

- добавить зависимости Supabase только на этом этапе реализации, не в текущей документационной задаче;
- описать `.env`-контракт: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, серверные ключи только для backend;
- создать `shared/api` или `shared/lib/supabase` для browser/server clients по текущим паттернам проекта;
- добавить базовые типы данных, соответствующие `Database.md`;
- оставить публичный UI на моках до готовности схемы.

Проверки:

- точечный type-check затронутых файлов;
- проверка, что секреты не попадают в клиентский bundle;
- ручная проверка, что публичный прототип не изменился.

Content-модель разделяет `organization_types` и `publication_categories`, а изображения привязывает через `media_assets`.

Работы:

- создать таблицы из `Database.md`;
- добавить foreign keys, check constraints и timestamps;
- добавить индексы для публичной ленты, каталога, кабинета и админ-панели;

- описать Storage buckets и policies для изображений организаций, заявок, публикаций и меню.

Проверки:

## 4. Этап 3. Публичное чтение из Supabase

Цель: заменить моковые данные публичной части на Supabase queries с сохранением текущего UI.

Проверки:

- проверка, что завершённые публикации не попадают в основную ленту;

## 6. Этап 5. Заявка организации и одобрение

Цель: реализовать путь от регистрации представителя до активной организации.

Статус: выполнено 2026-07-20. Реализованы форма заявки организации, сохранение черновика, отправка в `submitted`, админский список и карточка заявки, действия `approve`, `reject`, `request changes`, создание активной организации и owner membership при одобрении.

Работы:

- сделать пошаговую форму заявки по `Screen.md`;
- сохранять черновик заявки;
- отправлять заявку в статус `submitted`;
- добавить админский список заявок;
- реализовать действия `approve`, `reject`, `request changes`;
- при одобрении создавать или активировать организацию и owner membership.

Проверки:

- представитель создаёт и отправляет заявку;
- представитель видит статус своей заявки;
- администратор одобряет заявку;

- пользователь не может одобрить свою заявку без роли admin.

## 7. Этап 6. Кабинет организации

Цель: дать организации самостоятельно вести профиль, публикации и меню/услуги.

Статус: выполнено 2026-07-21. Добавлен защищённый кабинет активной организации с обзором, редактированием профиля через безопасный RPC, управлением публикациями, меню/услугами, базовыми показателями и просмотром представителей. Загрузка изображений не добавлялась: Storage upload UI не входит в этот этап.

Работы:

- обзор кабинета с минимальными показателями;
- редактирование профиля организации;
- создание и редактирование публикаций по шагам: тип, содержание, время, место и цена, предпросмотр, сохранение;
- статусы публикаций: черновик, запланировано, опубликовано, отменено, завершено;
- управление `menu_categories` и `menu_items`;
- управление представителями минимум на уровне просмотра, owner-действия добавлять только если нужны для MVP-экрана.

Проверки:

- представитель видит только свои организации;
- представитель не может изменить чужую организацию;
- представитель не может подменить `author_id`;
- черновик можно удалить;
- опубликованную публикацию можно отменить и завершить;
- меню организации отображается на публичной странице.

## 8. Этап 7. Модерация, важные объявления и неточности

Статус: выполнено 2026-07-21. Добавлены админские разделы публикаций, организаций, сообщений о неточности и важных объявлений, серверные действия модерации, публичная форма «Сообщить о неточности» в дополнительном меню публикации и безопасный RPC `create_inaccuracy_report` с проверкой публичности, дублей и частоты обращений.

Работы:

- админский список публикаций;
- скрытие и восстановление публикаций;
- блокировка организаций и публикаций;
- создание, активация и снятие важных объявлений;

Проверки:

## 9. Этап 8. Простая аналитика

Цель: добавить базовую статистику MVP без сложной аналитической системы.

Статус: выполнено 2026-07-21. Добавлена клиентская запись публичных событий `organization_view`, `publication_view`, `phone_click`, `route_click`, `menu_open`, `favorite_add`, агрегаты статистики для кабинета организации и компактная аналитика за 30 дней в админ-панели. Новая миграция не потребовалась: таблица `analytics_events`, индексы и RLS уже были добавлены на этапе 2.

Работы:

- записывать события `organization_view`, `publication_view`, `phone_click`, `route_click`, `menu_open`, `favorite_add`;
- связать события с `organization_id` и `publication_id`, когда они известны;
- добавить простые агрегаты для кабинета организации;
- добавить простые агрегаты для админ-панели при необходимости.

Проверки:

- события создаются без персональных данных сверх MVP-необходимости;
- представитель видит статистику только своих организаций;

## 10. Минимальная последовательность релиза

1. Сначала поднять схему, RLS и storage policies.
2. Затем подключить публичное чтение к существующим экранам.
3. После этого включить Auth и заявки.
4. Затем открыть кабинет организации.
5. В конце добавить админ-панель, неточности, важные объявления и аналитику.

Такой порядок снижает риск: публичная ценность приложения появляется раньше, а защищённые операции добавляются после того, как RLS уже проверена.

## 11. Что не делать в рамках backend MVP

- не добавлять оплату, заказы, корзину, билеты и бронирования;
- не добавлять публичные комментарии, отзывы и рейтинги;
- не добавлять чат;
- не делать сложные рекомендации;
- не синхронизировать избранное на сервере;
- не строить отдельное аналитическое хранилище;
- не заменять RLS проверками только в приложении;
- не использовать service role key в клиентском коде.

## 12. Следующие продуктовые итерации

### Этап 1. Находимость каталога

Статус: выполнено 2026-07-23. В публичном каталоге добавлены URL-фильтры: поиск по названию организации через `q` и выбор одного активного типа через `type`. Фильтрация выполняется серверными запросами Supabase и сохраняет публичные RLS-ограничения. Новые таблицы, RLS-политики, миграции и зависимости не добавлялись.

Входит:

- GET-форма с полем поиска, селектом активной категории и сбросом параметров;
- отдельное пустое состояние для запроса без результатов;
- нормализация URL-параметров и unit-тесты для неё;
- выборка только публикаций уже найденных организаций для счётчика активных публикаций.

Не входит:

- полнотекстовый поиск по описанию, адресу, меню или услугам;
- пагинация, персональные рекомендации и изменение карточек организаций.

Проверки:

- `npm test`;
- `npx tsc --noEmit`;

### Этап 3. URL-состояние фильтров городской ленты

Статус: выполнено 2026-07-23. Выбор одного фильтра городской ленты сохраняется в URL-параметре `filter`: «всё» остаётся состоянием по умолчанию на `/`, а остальные фильтры открываются по прямым ссылкам. Некорректные, пустые и повторённые параметры безопасно показывают всю ленту. Новые зависимости, запросы к БД, миграции, RLS-политики, аналитика и PWA-кеширование не добавлялись.

Входит:

- серверная нормализация одного URL-параметра `filter` по существующему набору фильтров;
- доступные ссылки-чипы с обозначением текущего фильтра и видимым keyboard focus;
- отдельное пустое состояние для активного фильтра без результатов и сбросом на `/`;
- unit-тесты нормализации `filter`.

Проверки:

- `npm test`;
- `npx tsc --noEmit`;
- `npm run lint`;
- ручная проверка фильтров, URL-навигации, пустых состояний, mobile и desktop.

### Этап 4. SEO и корректные превью публичных ссылок

Статус: выполнено 2026-07-23. Для главной, каталога, погоды, публичных страниц организации и публикации добавлены canonical URL, title, description и Open Graph metadata. Динамические metadata и JSON-LD читают только публичные данные анонимным Supabase-клиентом. JSON-LD `Organization` и `Event` строятся без персональных, выдуманных или непубличных данных; `Event` создаётся только для корректного временного диапазона. Private Storage и временные signed URL не используются в Open Graph.

Входит:

- безопасная нормализация локального и production origin через `NEXT_PUBLIC_SITE_URL`;
- metadata и корректные публичные canonical URL для scoped-маршрутов;
- стабильные внешние `https`-изображения для Open Graph только при наличии;
- unit-тесты metadata, JSON-LD, временного диапазона и экранирования текста.

Не входит:

- sitemap, robots, Twitter Cards и metadata избранного;
- изменение PWA-кеширования, UI, RLS, схемы БД, миграций и зависимостей.

Проверки:

- `npm test`;
- `npx tsc --noEmit`;
- `npm run lint`;
- ручная проверка HTML metadata для главной, организации и публикации.
